# 面试问答

## A. 项目整体与业务设计

### 1. 这个项目一句话是什么？

这是一个基于 LangGraph 的电商售后客服 Agent 个人项目，用模拟政策、订单和工单数据实现政策问答、订单查询、退款判断、工单确认和人工处理建议。

### 2. 项目为什么选售后场景？

售后场景天然有工具边界：政策解释适合 RAG，订单查询适合数据库，退款资格适合规则，风险问题适合人工处理，适合展示 Agent 编排能力。

### 3. 当前是真实业务系统吗？

不是。它是面试展示级工程化个人项目，使用模拟政策和模拟订单，不接入真实电商平台，也不执行真实退款或售后动作。

### 4. 核心能力有哪些？

核心能力包括政策咨询、订单查询、退款资格判断、模拟工单创建、高风险问题人工处理建议，以及 FastAPI 服务化、评测和稳定性测试。

### 5. 你怎么划分业务边界？

通用政策走 RAG，具体订单事实走 SQLite，退款结论走规则引擎，写工单必须确认，账户安全和资金争议转人工建议。

### 6. 为什么不直接让 LLM 回答所有问题？

因为订单状态和退款资格需要稳定、可复现、可测试。LLM 可以解释政策，但不能替代订单事实查询和确定性规则判断。

### 7. 项目的完成度如何？

已完成本地服务化、RAG、LangGraph 编排、API、日志、评测、稳定性测试和 Dockerfile。未实现多 Agent、Memory、鉴权、高可用和生产部署。

### 8. 这个项目最重要的工程点是什么？

核心是把 LLM 能力限制在合适边界内，用 Tool、规则、状态图、确认机制和评测保证系统行为可控。

## B. RAG 与检索

### 9. BM25、Dense、Hybrid 有什么差异？

BM25 依赖关键词匹配，Dense 依赖向量语义相似，Hybrid 用 RRF 融合两者排名。它们各有优势，不存在永远最优。

### 10. 为什么 Hybrid 没有必然更好？

当前政策库小且业务术语明确，BM25 Top3 覆盖最好。Hybrid 可能把语义相近但业务阶段不直接的政策排上来。

### 11. 为什么默认 BM25 Top3？

实验结果显示 BM25 Hit@3 为 1.0，召回稳定且可解释。对当前模拟政策库，默认 BM25 Top3 是更保守的选择。

### 12. BGE 和 FAISS 的作用是什么？

BGE 把中文政策文本编码成向量，FAISS 用 IndexFlatIP 做相似向量检索。当前规模小，所以使用精确检索而不是复杂近似索引。

### 13. 向量相似为什么不等于业务相关？

语义相近可能跨业务阶段，例如退款、取消、拒收相互接近，但用户真正需要的政策可能属于其中一个更具体阶段。

### 14. has_relevant_policy 有什么意义？

它用于判断检索结果是否足以回答问题，避免天气、写诗等无关问题被强行回答成售后政策。

### 15. Policy QA 链路是什么？

先检索政策 chunk，再构造证据包，LLM 输出 JSON，最后校验引用是否来自本次 retrieved_chunks，形成带引用回答。

### 16. 引用校验做了什么？

引用校验检查模型给出的 cited_chunk_ids 是否来自本次检索证据，防止引用不存在或未检索到的材料。

### 17. 引用校验为什么不能完全防幻觉？

它只能证明引用合法，不能证明模型对证据的解释完全正确。语义正确性仍需要评测、badcase 和人工抽检。

### 18. 真实 Policy QA 验收结果如何？

固定 6 条真实模型验收全部通过，但这只说明当前验收集通过，不代表覆盖所有开放问题。

## C. Agent 与 LangGraph

### 19. 为什么用 LangGraph？

LangGraph 适合表达受控状态流和条件边。售后场景需要根据意图、订单号和确认标志走不同分支，用图比一个大函数更清晰。

### 20. State 保存什么？

State 保存单次请求的用户问题、订单号、意图、工具结果、回答状态、引用、错误、debug 和 tool_trace。它不是长期 Memory。

### 21. 有哪些节点？

主要节点包括接收请求、实体提取、意图分类、缺订单号、政策问答、订单查询、退款判断、工单确认、创建工单、人工处理和格式化响应。

### 22. 规则路由有什么局限？

规则路由对当前受控任务稳定可测，但对复杂口语、多轮上下文和隐含意图泛化有限，后续可尝试 LLM 路由加规则兜底。

### 23. 为什么不使用多 Agent？

当前任务边界清楚，单 Agent 加受控 Tool 足够。多 Agent 会增加通信和调试复杂度，不适合这个阶段。

### 24. 缺订单号怎么处理？

如果订单查询或退款判断缺订单号，系统返回 `missing_order_id`，提示提供 `ORD10003` 格式订单号，不调用订单或退款 Tool。

### 25. unknown 怎么处理？

天气、写诗等不属于当前售后范围的问题走 `unknown_request`，不调用 LLM 和业务 Tool。

## D. Tool Calling 与规则引擎

### 26. 为什么不用 LLM 直接判断退款？

退款资格依赖订单事实和明确规则，必须稳定、可复现、可测试。LLM 只负责政策解释，不负责退款结论。

### 27. 退款规则考虑了哪些事实？

包括支付状态、发货状态、签收时间、退款状态、是否拆封、是否存在质量问题，以及异常字段或未知状态。

### 28. Tool Calling 的边界是什么？

Graph 只负责选择 Tool，不复制 SQL、退款规则或 RAG 逻辑。具体业务能力放在 `app/tools` 和 `app/services`。

### 29. 如何防止 Agent 自动创建工单？

创建工单需要 `confirm_ticket_creation=True`。未确认时只返回 `ticket_confirmation_required`，不会写数据库。

### 30. 高风险问题为什么不自动处理？

重复扣款、账号安全、隐私投诉可能涉及敏感信息和资金争议，当前系统只给人工处理建议，不调用 LLM、不写数据库。

### 31. 订单查询会泄露用户数据吗？

响应只返回必要状态字段，如订单号、商品名、支付、物流和退款状态，不返回不必要的用户数据。

### 32. 订单不存在怎么处理？

订单查询会提示核对订单号；退款资格 Tool 对不存在订单返回 `manual_review` 和 `ORDER_NOT_FOUND` 结构化结果。

## E. FastAPI、日志、测试与稳定性

### 33. FastAPI 层做什么？

FastAPI 负责请求校验、request_id、Header、依赖注入、并发保护、超时、安全错误响应和结构化日志。

### 34. request_id 有什么作用？

request_id 用于串联一次 API 请求、Agent 执行、响应 Header 和日志，方便排查问题。

### 35. API 如何隔离真实 LLM？

`get_policy_qa_callable()` 可被 FastAPI dependency override 替换。pytest 中使用 stub，避免真实调用 DeepSeek。

### 36. 结构化日志记录什么？

日志记录 endpoint、http_status、intent、tool_used、answer_status、latency、error_code、query_length 和脱敏订单号，不记录完整 API Key 或完整问题。

### 37. 工具耗时怎么统计？

真实 Tool 调用通过 `time.perf_counter()` 包裹，记录 `latency_ms`，并在响应中汇总 `tool_latency_summary`。

### 38. 如何做 API 验收？

`run_api_checks.py` 会真实启动 Uvicorn，调用 health、订单查询、退款判断、缺订单号、工单确认、人工处理和 unknown，不测试真实 LLM。

### 39. 稳定性测试测了什么？

它测试 12 次顺序调用、5 个并发请求、request_id 唯一性、未预期 5xx、API 耗时和本次追加日志安全。

### 40. 为什么 503 agent_busy 是合理结果？

当前有进程内并发保护。并发超过上限时返回结构化 `agent_busy`，比请求堆积或未处理异常更可控。

### 41. pytest 覆盖哪些层？

覆盖检索、Policy QA、引用校验、退款规则、订单 Tool、Agent、API、bootstrap 和稳定性 helper。当前全量结果为 87 passed。

### 42. run_full_checks 做什么？

它按顺序运行初始化、构建索引、pytest、API 验收和稳定性测试，并输出 summary 和 report。

## F. Badcase 与工程边界

### 43. 项目最真实的 badcase 是什么？

典型 badcase 是通用保修问题曾被过度标记为 manual_review，以及已发货取消问题曾优先引用退款政策而非取消政策。

### 44. Q1、Q3 是怎么修复的？

通过政策切块优化和 Prompt 约束，增加更直接的保修、已发货取消/拒收证据。真实 LLM 验收从 4/6 提升到 6/6。

### 45. Dense / Hybrid 的主要 badcase 是什么？

语义相近政策之间会排序偏差，例如退款政策压过取消政策，或服务范围问题被退款政策吸引。

### 46. Docker 为什么还不能说已部署？

当前只生成 Dockerfile，环境未检测到 Docker CLI，未完成镜像构建和容器健康检查验证。配置不等于生产部署。

### 47. 为什么不说模型准确率 100%？

1.0 指标来自受控评测集，只证明当前测试集表现，不代表开放真实场景泛化能力。

### 48. 当前最需要补的是什么？

更大的独立评测集、更多真实口语 badcase、容器实际构建验证、API 鉴权限流和监控体系。
