# Badcase 分析

本文基于已有真实结果文件整理，不代表系统覆盖所有真实开放问题。

## Badcase 1：保修问题被过度标记为 manual_review

现象：用户问“耳机买了以后一般保修多长时间？”，早期真实 LLM 验收中检索命中保修政策，但模型把“实际申请保修可能需要凭证”理解成当前问题必须人工处理。

影响：通用政策解释被过度升级为人工处理，降低自动回答覆盖率。

根因：证据包中直接说明“12 个月保修”的 chunk 不够突出，Prompt 对“政策解释”和“订单级执行材料”的边界约束不够强。

修复方式：新增“数码商品与耳机保修期限”小节，并强化 Prompt：通用政策已有直接证据时应回答，不因后续执行材料要求而直接 manual_review。

修复后验证：Q1 在真实 Policy QA 验收中通过，状态为 `answered`，引用 `invoice_warranty_policy__section_03`。

仍然存在的边界：这只覆盖当前固定验收问题，不代表所有保修相关口语问法都能正确处理。

## Badcase 2：已发货取消问题优先引用了退款政策而不是取消政策

现象：用户问“商品已经发货，但我临时不想要了怎么办？”，早期回答引用了较泛化的退款政策，而不是更直接的取消/拒收政策。

影响：答案引用虽然可能相关，但业务阶段不够精确，面试中会暴露 RAG 证据选择偏差。

根因：已发货、拒收、取消、退款在语义上相近，原取消政策中直接覆盖已发货阶段的 chunk 不够突出。

修复方式：在取消政策中新增“已发货订单的取消、拒收与售后处理”小节，并要求优先引用覆盖用户核心动作和业务阶段的证据。

修复后验证：Q3 在真实 Policy QA 验收中通过，引用 `cancellation_policy__section_03`。

仍然存在的边界：如果用户表达更绕，Dense / Hybrid 仍可能把退款政策排在更前。

## Badcase 3：Dense / Hybrid 在语义相近政策之间排序偏差

现象：Dense 和 Hybrid badcase 中，多次出现退款政策压过取消政策、物流政策或服务范围政策。例如“快递已经发出但还没签收，我可以直接退款吗？”Top1 为退款政策，期望来源是取消政策。

影响：Top3 有时能覆盖期望来源，但 Top1 不稳定会影响生成模型优先关注的证据。

根因：向量相似捕捉的是语义接近，不等于业务阶段和处理路径完全匹配。售后政策中“退款、取消、拒收、换货、物流异常”语义高度交叉。

修复方式：继续优化政策切块、增加业务阶段关键词、校准相关性阈值，并可在 rerank 阶段引入“业务阶段匹配”特征。

修复后验证：当前检索对比中 BM25 Top3 为 1.0000，Dense Top3 为 0.8667，Hybrid Top3 为 0.9333；说明混合检索有帮助但不是绝对最优。

仍然存在的边界：当前没有训练 reranker，也没有大规模开发集校准阈值。

## 修复效果说明

Q1、Q3 经政策切块优化和 Prompt 约束后，真实 LLM 验收由 4/6 提升到 6/6。这不表示系统已经覆盖所有真实开放问题。

## 复合任务边界记录：退款判断后再创建工单

新增 `refund_then_ticket_if_ineligible` 后，系统可以处理“先判断退款资格，如果不能退款再帮我创建售后工单”的固定复合任务。

已修复风险：该场景不会直接写工单；只有退款资格结果为 `not_eligible` 且用户明确要求兜底工单时，才保存 `pending_action=create_ticket` 并等待同一 `session_id` 的第二轮确认。

仍保留边界：ASAE007、ASAE016 等真实 badcase 没有为了指标被删除。当前复合任务不是通用任务规划器，不支持任意多步骤售后流程，也不代表 LLM 可以自主决定调用写操作工具。
