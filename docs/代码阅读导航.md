# 代码阅读导航

这份文档用于学习项目代码，不属于运行时功能。它按“先理解业务，再理解 Agent，再理解 Tool，最后理解工程化能力”的顺序组织。

## 1. 推荐阅读顺序

### 第一轮：先理解业务

1. `README.md`
2. `docs/PROJECT_SUMMARY.md`
3. `docs/PROJECT_ARCHITECTURE.md`
4. `docs/DEMO_GUIDE.md`

这一轮先回答：项目解决什么问题、哪些能力已经完成、哪些边界不能越过。

### 第二轮：理解 Agent 主流程

1. `app/agent/after_sales_agent_service.py`
2. `app/agent/agent_state.py`
3. `app/agent/intent_classifier.py`
4. `app/agent/after_sales_graph.py`
5. `app/agent/response_formatter.py`

这一轮重点看一次用户请求如何变成 State，如何分类、路由、调用 Tool，最后如何变成统一 JSON。

### 第三轮：理解各类 Tool

1. `app/tools/order_tool.py`
2. `app/services/refund_rule_engine.py`
3. `app/tools/refund_eligibility_tool.py`
4. `app/services/policy_qa_service.py`
5. `app/tools/policy_qa_tool.py`

这一轮重点区分：谁读数据库、谁写数据库、谁调用 LLM、谁只做纯规则判断。

### 第四轮：理解工程化能力

1. `app/api_server.py`
2. `app/observability/logging_config.py`
3. `scripts/run_api_checks.py`
4. `scripts/stability_test.py`
5. `scripts/run_full_checks.py`

这一轮重点看服务化、request_id、结构化日志、API 验收、稳定性测试和全量回归。

## 4. 初学者最容易混淆的概念

### Tool

Tool 是 Agent 能调用的业务能力包装。它通常有清晰输入输出，比如订单号进来、订单事实出去。Tool 不应该变成大杂烩，也不应该在图里重复实现 SQL 或规则。

### Service

Service 是更底层的业务服务。比如退款规则引擎和 Policy QA 服务都在 `app/services/`。Tool 可以调用 Service，把它包装成 Agent 更容易使用的接口。

### Retriever

Retriever 负责从政策库里找候选证据。它不负责最终回答，只负责“找材料”。本项目有 BM25、FAISS 和 Hybrid 检索。

### Rule Engine

Rule Engine 是规则引擎。退款资格依赖订单状态、签收时间、拆封和质量问题，所以用规则稳定判断，不让 LLM 猜。

### State

State 是 LangGraph 中一次请求的临时状态。它记录 user_query、order_id、intent、tool_result、answer_status 等。它不是长期记忆，也不保存 API Key。

### Node

Node 是图里的一个步骤。每个节点只做一件事，比如提取订单号、分类意图、调用退款 Tool 或格式化响应。

### Conditional Edge

Conditional Edge 是条件边。它根据当前 State 决定下一个节点，比如缺订单号就追问，有订单号就查订单。

### Schema

Schema 用来约束数据结构。API 请求、API 响应和 Agent 响应都用 schema 让字段稳定，方便测试和调用。

### Middleware

Middleware 是请求进入路由前后执行的逻辑。本项目用它生成或透传 request_id，并把 request_id 写回响应 Header。

### request_id

request_id 是一次请求的追踪编号。它出现在请求状态、响应 body、响应 Header 和日志中，方便排查问题。

### Verifier

Verifier 是校验器。引用校验器检查模型引用的 chunk_id 是否来自本次检索证据，但不能保证答案语义绝对正确。

### RAG

RAG 是先检索再生成。模型不是凭空回答，而是基于检索到的政策 chunk 生成答案。

### BM25

BM25 是关键词检索。它很适合售后政策里的明确词，比如退款、发票、保修，但对同义表达理解有限。

### Embedding

Embedding 把文本变成向量，让相近语义有相近向量。它能处理部分同义表达，但向量相似不等于业务正确。

### FAISS

FAISS 是向量检索库。本项目用 `IndexFlatIP` 做精确内积搜索，因为政策 chunk 数量小，不需要复杂近似索引。

### RRF

RRF 是排名融合方法。它融合 BM25 和 Dense 的排名，不直接相加不同量纲的分数。
