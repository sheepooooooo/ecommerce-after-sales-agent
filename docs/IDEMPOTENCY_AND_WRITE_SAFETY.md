# 工单创建幂等与写安全说明

本文档说明当前售后 Agent 对“创建模拟工单”这一写操作的幂等保护。所有工单均写入本地 SQLite 模拟数据库，不执行真实退款、真实取消或真实客服流转。

## 什么是幂等

幂等是指同一个写操作被重复提交时，系统不会重复产生副作用。例如，同一个待确认工单操作重复确认时，应返回同一个 `ticket_id`，而不是创建多张工单。

## 为什么需要幂等保护

以下情况都可能导致重复写入：

- 用户连续点击“确认”；
- 浏览器或客户端超时后重试；
- API 调用方重复发送同一请求；
- 网络抖动导致调用方无法确认第一次请求是否成功。

如果没有幂等保护，重复请求可能创建多张售后工单，影响演示数据和评测结果。

## 本项目如何生成和绑定 key

- API 可传入 `idempotency_key`；
- 如果调用方未传，服务端会生成一个随机 key；
- 多轮确认时，第一轮创建待确认动作会把 key 保存到 `conversation_sessions.pending_payload`；
- 第二轮用户输入“确认”时，服务端从同一 `session_id` 的 pending action 中恢复该 key；
- 真正创建工单前，会先写入或读取 `ticket_idempotency_records` 表；
- 若相同 key 已完成，则直接返回已有 `ticket_id`，不再次调用真实写入逻辑。

## 数据库约束

`ticket_idempotency_records.idempotency_key` 是主键。它提供数据库层唯一约束，避免只靠 Python 内存判断。

## 幂等不等于失败重试

当前阶段没有加入自动重试。幂等保护只负责防止重复执行已确认的同一写操作，不负责在工具失败时自动再次写入。

## 当前局限

- 当前为单机 SQLite，不是分布式事务；
- 没有真实用户身份认证；
- 没有跨设备权限校验；
- 没有幂等 key 过期清理策略；
- 不应宣称 production-ready；
- 只保护本地模拟工单创建，不涉及真实支付、退款或物流操作。
